# 01_사이트맵_전환설계

> 저장 위치: 리포지토리 `docs/기획/01_사이트맵_전환설계.md`
> 전제: 00_핵심방향요약의 방향을 따른다. 블로그(주 1회 발행)는 "정비 사례"로 통합되어 새 사이트에 네이티브로 쌓인다. 정적 사이트(Astro + Cloudflare Pages)이므로 폼 백엔드가 없고, 전환은 전화 탭 투 콜과 네이버 예약을 축으로 한다.

---

## 1. 페이지 구성도 (사이트맵)

```
메인(홈) /
 ├─ 서비스 안내        /services/
 │    └─ (증상별 + 정비 항목별 섹션 — 단일 페이지 내 앵커 구성)
 ├─ 정비 사례 (=블로그)  /cases/            ← 주 1회 발행, 사이트의 SEO·GEO 자산
 │    ├─ 목록 (전체 / 차종별 / 증상별 필터)
 │    └─ 개별 사례       /cases/{slug}/     ← 차종→증상→원인→수리→결과→사진 고정 포맷
 ├─ 진단 장비 & 소개    /equipment/
 │    └─ (장비 풀라인 + 대표 20년 경력 + ISO 9001·14001·45001)
 ├─ 오시는 길          /location/
 │    └─ (지도·주소·영업시간·서비스 지역: 동탄·화성·오산·평택)
 └─ 문의 / 예약        /contact/
      └─ (전화 탭 투 콜 · 네이버 예약 링크 · 첫 방문 안내)
```

**구성 원칙:**
- 정비 사례가 곧 블로그다. 별도 블로그 메뉴를 두지 않는다. 모든 주간 글은 정비 사례 포맷(차종→증상→원인→수리→결과→사진)으로 `/cases/`에 누적된다.
- 네이버 블로그(blog.naver.com/extran)는 기존 신뢰 채널로 유지하되, 새 사이트 사례는 새로 시작한다(마이그레이션 없음).
- URL 슬러그는 영문 소문자·하이픈 기준(예: `/cases/bmw-520d-coolant-leak/`). 상세 규칙은 page_정비사례.md 단계에서 확정한다.
- 서비스 안내는 초기엔 단일 페이지 + 증상별 앵커로 시작하고, 사례가 쌓여 브랜드별·증상별 트래픽이 확인되면 하위 페이지로 분리한다(정적 사이트라 확장 비용 낮음).

## 2. 네비게이션 메뉴

**메뉴 구성(6개):** 홈 · 서비스 · 정비 사례 · 진단 장비 · 오시는 길 · 문의

**데스크톱 헤더:**
```
[가로형 로고(좌)] [홈 서비스 정비사례 진단장비 오시는길 문의] [전화 상담 버튼 + 010-3203-7607(우)]
```
- 우측 전화 버튼은 단일 블루 액센트 컬러(디자인 토큰의 액센트)로 헤더 내 유일한 강조 요소.

**모바일 헤더:**
```
[가로형 로고(좌)] [전화 아이콘(tel: 탭 투 콜)] [햄버거 메뉴(우)]
```
- 햄버거 펼침 시: 메뉴 6개 + 하단에 "전화로 증상 상담하기" · "네이버로 예약하기" 버튼 2개.

**푸터 (전 페이지 공통):**
- 상호(모터리페어 / 장용석의 내차를 부탁해 동탄신도시점) · 사업자등록번호 289-02-02283
- 주소(경기도 화성시 효행로1156번길 9) · 영업시간 · 전화(tel: 링크) · 네이버 예약 링크
- ISO 9001·14001·45001 취득 표기 · 원형 엠블럼(파비콘과 동일 심볼)
- 서비스 지역 텍스트: "동탄·화성·오산·평택 수입차 정비" (로컬 SEO용 텍스트 명시)

## 3. 전환 설계

**전환 계층:**

| 구분 | 액션 | CTA 문구 | 배치 위치 |
|---|---|---|---|
| 주 전환 ① | 전화 상담 (tel: 010-3203-7607) | "전화로 증상 상담하기" | 히어로 · 각 페이지 하단 · 플로팅 · 헤더 |
| 주 전환 ② | 네이버 예약 (스마트플레이스, 새 창) | "네이버로 예약하기" | 히어로 · 각 페이지 하단 · 플로팅 · 문의 페이지 |
| 보조 전환 | 정비 사례 열람 | "비슷한 사례 보기" | 메인 중단 · 서비스 각 항목 하단 |
| 보조 전환 | 진단 장비 확인 | "어떤 장비로 진단하는지 보기" | 메인 · 서비스 |

**정적 사이트 제약 반영:**
- 자체 문의 폼 백엔드가 없다. 문의/예약 페이지는 입력 폼 대신 ① 전화 탭 투 콜 대형 버튼, ② 네이버 예약 버튼, ③ 첫 방문 안내(준비물·절차)로 구성한다.
- 향후 폼이 필요해지면 Cloudflare Pages Functions 또는 외부 폼 서비스 연결을 별도 과제로 둔다(초기 범위 밖).

**가격 정책:** "작업 전 견적 안내" 방식으로 통일. 가격표를 두지 않고, 가격 관련 영역은 견적 상담 CTA로 대체한다.

**모바일 플로팅 CTA:**
- 우측 하단 고정. 전화(탭 투 콜)와 네이버 예약 2개이므로 FAB(플로팅 액션 버튼 — 탭하면 펼쳐지는 원형 버튼)로 묶어 펼침 처리.
- 버튼 높이 44px 이상, 버튼 간 간격 8px 이상 (모바일 설계 원칙 준수).
- 정비 사례 개별 글 하단에는 플로팅과 별개로 "이런 증상이면 전화 주세요" 인라인 CTA를 고정 배치(글 읽고 온 검색 유입의 전환 지점).

## 4. 방문자 유형별 동선

| 유형 | 진입 | 동선 | 각 단계에서 해소할 불안 |
|---|---|---|---|
| ① 비교형 (핵심 전환 타깃) | 검색 → 메인 또는 정비 사례 개별 글 | 메인 → 진단 장비/소개 → 정비 사례 → 서비스 → 문의 | 진단 장비·경력·ISO(실력 검증) → 사례 사진(바가지 우려) → A/S 명시(보증 불안) |
| ② 확인형 (증상 발생·급함) | 네이버 플레이스 / 메인 직행 | 메인 → 전화 직행 / 메인 → 서비스(증상 앵커) → 전화 | 히어로에서 3초 안에 업종·지역 확인 → 증상 언어("경고등이 떴어요")로 바로 매칭 → 원탭 전화 |
| ③ 단골·재방문 | 직접 방문 / 즐겨찾기 | 메인 → 푸터(연락처) / 네이버 예약 직행 | 불안 없음 — 연락처·예약이 어디서든 1탭 거리 |

**검색 유입 특수 동선 (사례 글 직접 진입):**
주간 사례가 쌓이면 "차종+증상" 검색으로 개별 사례 글에 바로 떨어지는 방문자가 늘어난다. 이 동선을 위해 사례 상세 페이지는 독립적으로 전환이 완결되어야 한다:
```
검색("BMW 520d 냉각수 누수") → /cases/{slug}/
  → 글 하단: "비슷한 증상이신가요? 전화로 증상 상담하기" CTA
  → 관련 사례 2~3개 카드 → 진단 장비 페이지 링크
```

## 5. 콘텐츠 발행 운영 흐름 (사이트맵에 미치는 영향)

- 주 1회 발행: 사진 + Gemini 생성 원고 → Claude Code `publish-case` 명령 → 캡션 검토 → push → Cloudflare 자동 배포.
- 발행할수록 `/cases/` 목록이 성장하며, 메인의 "최근 정비 사례" 섹션(3~6개 카드)이 자동 갱신되는 구조로 설계한다(Astro 콘텐츠 컬렉션 최신순 쿼리).
- 사례 수 12편 이상 누적 시 차종별·증상별 필터(태그 기반 정적 페이지) 활성화를 검토한다.
