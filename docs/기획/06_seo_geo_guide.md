## 3. H 구조 규칙

- H1은 페이지당 1개 (각 page_*.md의 히어로/헤드).
- H2 = 섹션 제목. 증상·질문형 H2 적극 사용 ("경고등이 떴어요 · 원인을 모르겠어요") — 검색 질의와 문면 일치.
- H3는 사례 본문 내부에서만 (젬 본문 구조 따름). 장식용 헤딩 금지.

## 4. GEO 전략 (AI 검색 노출)

**원칙: "한 문장만 떼어도 사실이 성립"하는 자기완결 문장을 핵심 위치에 박는다.**

| 장치 | 위치 | 상태 |
|---|---|---|
| 엔티티 선언 블록 | 메인 §3 첫 문장 ("모터리페어는 경기 화성 동탄에서 …") | 반영됨 |
| 자기완결 서비스 문장 | 진단장비 §2 카드 4개 ("모터리페어는 ~을 보유하고 ~에 사용합니다") | 반영됨 |
| 자연어 FAQ | 문의 8문항 + 메인 3문항 + 사례별 frontmatter faq 3문항, 전부 펼친 상태 | 반영됨 |
| 사례 표준 포맷 | 차종→증상→원인→수리→결과→사진 + summary_lead 요약 카드 | 파이프라인 반영됨 |
| 신뢰 근거 텍스트화 | "ISO 9001·14001·45001 취득, 20년 경력, A/S 1년/2만km" — 이미지 아닌 텍스트 | 반영됨 |
| symptom_customer | 고객 언어 증상("뒤가 주저앉아요")을 사례 메타+전환 블록에 노출 | 파이프라인 반영됨 |
| 구조화 데이터 | AutoRepair(전 페이지) + FAQPage(문의) | 05 문서 정의 |

**작성 규칙 (매주 사례 발행 시 젬·검토 단계에서 유지):**
- 5W1H 명시 (어디서·무엇을·어떻게·결과).
- 수치(주행거리·부품명·기간)는 이미지가 아닌 텍스트로.
- 단계 구조 유지 (포맷 고정이 곧 GEO 자산).
- "판단됩니다" 등 확신 서술, "대략·약" 금지 (젬 지침과 일치).

## 5. 기술 SEO 체크리스트 (Astro 빌드 시)

- [ ] sitemap.xml 자동 생성 (@astrojs/sitemap) + robots.txt
- [ ] 페이지별 canonical URL
- [ ] OG 태그: 메인=외관 사진 / 사례=thumbnail (og:image 1200×630 자동 생성)
- [ ] 이미지 alt 전수 적용 ([지역+업종+설명] / 장식 alt="")
- [ ] Lighthouse 모바일 성능 90+ 목표 (정적+이미지 최적화로 달성 가능)
- [ ] 오픈 후: 네이버 서치어드바이저 + 구글 서치콘솔 등록, sitemap 제출

## 6. 정비 사례 권장 주제 로드맵 (초기 12주)

원칙: 실제 입고 차량이 소재다 — 아래는 우선 발행 순위 판단 기준이지 창작 목록이 아니다.

**우선 발행 기준 (검색 수요 × 차별화 입증):**
1. 진단 스토리가 있는 건 (경고등 → 원인 특정) — Picoscope·전용 진단기 활약 건 최우선
2. 브랜드 분산 — 초기 12편에 BMW·벤츠·아우디·포르쉐가 고루 들어가게 (브랜드별 검색 커버)
3. 계절 증상 선반영 — 여름: 에어컨·냉각수 / 겨울: 시동·배터리 (검색 수요 피크 2~4주 전 발행)
4. 대표 차종 우선 — 5시리즈·E클래스 등 등록 대수 많은 차종 (롱테일 모수가 큼)
- 재료 확보분: 벤츠 E400 에어쇼바 · BMW X7 디퍼렌셜 오일 (샘플 2편 — 파이프라인 검증 겸 첫 발행 후보)
